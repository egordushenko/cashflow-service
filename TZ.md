Нужно реализовать полноценное Django web-приложение по тестовому заданию: сервис для учёта движения денежных средств.

Цель:
Сделать рабочий веб-сервис, где пользователь может создавать, просматривать, редактировать и удалять записи о движении денежных средств, а также управлять справочниками: статусы, типы операций, категории и подкатегории. В приложении должны соблюдаться логические зависимости между сущностями.

Стек:
- Python 3.11+
- Django 5.x
- SQLite по умолчанию
- Django ORM
- Django Templates
- Bootstrap 5 для интерфейса
- Минимальный JavaScript только для зависимых select-полей
- Django REST Framework добавить минимально, если это не усложнит архитектуру
- Без React/Vue
- Без авторизации, если она не требуется по ТЗ
- Без сложного production-деплоя

Основные сущности:

1. Status
Поля:
- id
- name: string, unique

Примеры:
- Бизнес
- Личное
- Налог

2. OperationType
Поля:
- id
- name: string, unique

Примеры:
- Пополнение
- Списание

3. Category
Поля:
- id
- name: string
- operation_type: FK -> OperationType

Ограничения:
- category должна быть привязана к конкретному типу операции
- unique по паре name + operation_type

Примеры:
Для типа "Списание":
- Маркетинг
- Инфраструктура
- Зарплаты

Для типа "Пополнение":
- Продажи
- Возврат
- Инвестиции

4. Subcategory
Поля:
- id
- name: string
- category: FK -> Category

Ограничения:
- subcategory должна быть привязана к конкретной категории
- unique по паре name + category

Примеры:
Для категории "Инфраструктура":
- VPS
- Proxy
- Домены

Для категории "Маркетинг":
- Farpost
- Avito
- Telegram Ads

5. CashFlowRecord
Поля:
- id
- created_date: date
  - по умолчанию текущая дата
  - пользователь может изменить вручную
- status: FK -> Status
- operation_type: FK -> OperationType
- category: FK -> Category
- subcategory: FK -> Subcategory
- amount: DecimalField
  - сумма в рублях
  - обязательное поле
  - значение должно быть больше 0
- comment: TextField
  - необязательное поле

Бизнес-правила:
1. Пользователь не может выбрать категорию, которая не относится к выбранному типу операции.
2. Пользователь не может выбрать подкатегорию, которая не относится к выбранной категории.
3. Поля amount, operation_type, category, subcategory обязательны.
4. Валидация должна быть и на клиенте, и на сервере.
5. Если через POST-запрос передана неправильная связка type/category/subcategory, сервер должен вернуть ошибку формы.
6. При удалении справочников:
   - если справочник используется в записях ДДС или связанных сущностях, лучше не удалять его, а показывать понятную ошибку
   - не делать silent cascade deletion денежных записей

Страницы приложения:

1. Главная страница `/`
Назначение:
- отображает таблицу всех записей ДДС

Таблица должна содержать колонки:
- дата
- статус
- тип
- категория
- подкатегория
- сумма
- комментарий
- действия: редактировать / удалить

Фильтры:
- период дат: date_from, date_to
- статус
- тип
- категория
- подкатегория

Требования к фильтрам:
- фильтры должны работать через GET query params
- можно комбинировать несколько фильтров
- при выборе типа список категорий должен логически соответствовать типу
- при выборе категории список подкатегорий должен соответствовать категории
- после применения фильтра значения в форме должны сохраняться

2. Страница создания записи `/records/create/`
Форма:
- дата
- статус
- тип
- категория
- подкатегория
- сумма
- комментарий

Поведение:
- дата по умолчанию сегодняшняя
- категория выбирается только после выбора типа
- подкатегория выбирается только после выбора категории
- при ошибке форма возвращается с сообщениями ошибок

3. Страница редактирования записи `/records/<id>/edit/`
Те же поля, что и при создании.
Должна корректно подставлять текущие значения.
Зависимые select-поля должны работать и на уже существующей записи.

4. Удаление записи `/records/<id>/delete/`
Сделать страницу подтверждения удаления.
После удаления вернуть пользователя на главную.

5. Страница справочников `/references/`
На одной странице сделать управление:
- статусами
- типами операций
- категориями
- подкатегориями

Можно сделать простую структуру:
- 4 блока на одной странице
- в каждом блоке таблица существующих значений
- рядом/сверху форма добавления
- ссылки редактировать / удалить

Для категорий в форме обязательно выбирать тип операции.
Для подкатегорий в форме обязательно выбирать категорию.

6. CRUD для справочников
Нужны маршруты:
- create/edit/delete для Status
- create/edit/delete для OperationType
- create/edit/delete для Category
- create/edit/delete для Subcategory

Можно реализовать через class-based views или function-based views, главное — чисто и понятно.

AJAX / JSON endpoints:

Нужны endpoints для зависимых списков:

1. `/ajax/categories/?operation_type=<id>`
Возвращает JSON:
[
  {"id": 1, "name": "Маркетинг"},
  {"id": 2, "name": "Инфраструктура"}
]

2. `/ajax/subcategories/?category=<id>`
Возвращает JSON:
[
  {"id": 1, "name": "VPS"},
  {"id": 2, "name": "Proxy"}
]

Эти endpoints использовать в формах создания/редактирования записи и, желательно, в фильтрах.
